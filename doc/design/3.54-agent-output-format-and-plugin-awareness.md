# 3.54 Agent Output 格式修复与插件感知增强

## 问题背景

用户要求 Agent 在阿里云和 AWS 各开 2 台代理并生成 clash 配置，结果：
1. Agent 拿到的 IP 列表格式不可读，只生成了 2 台而非 4 台的 clash 配置
2. Agent 不知道 proxy 模板有 redc-plugin-clash-config 插件会自动生成 clash 配置，选择手动组装

## 根因分析

### 问题 1：get_case_outputs 返回 Go 原始结构体格式

**现象**：
```
| ecs_ip | {%!s(bool=false) ["tuple",["string","string"]] ["39.102.103.129","39.102.84.182"]} |
```

**原因**：`toolGetCaseOutputs` (mcp.go) 用 `fmt.Sprintf("| %s | %s |", k, v)` 格式化 `tfexec.OutputMeta` 结构体，Go 的 `%s` 对非 string 类型使用 `%v` 回退，输出原始结构体。

`OutputMeta` 结构：
```go
type OutputMeta struct {
    Sensitive bool            `json:"sensitive"`
    Type      json.RawMessage `json:"type"`
    Value     json.RawMessage `json:"value"`
}
```

`Value` 是 `json.RawMessage`（即 `[]byte`），直接 `%s` 格式化会输出 `{Sensitive Type Value}` 的 Go 内部表示。

**对比**：`app_scene.go` 的 `GetCaseOutputs()` 已正确处理——`string(meta.Value)` 提取 JSON 字符串并去引号。MCP 版本没有做同样处理。

### 问题 2：Agent 不知道插件系统

**现象**：Agent 读了 deploy.sh，试图理解 clash 配置生成逻辑，最终自己用 `curl ifconfig.me` 拿 IP 手动组装 clash YAML。

**原因**：
- proxy 模板的 case.json 声明了 `"redc_plugins": "redc-plugin-clash-config,redc-plugin-upload-r2"`
- RedC 在 start_case 的 post-apply hook 中自动执行插件，生成 `plugin_outputs.json`（含 `clash_config_file` 路径）
- `app_scene.go:GetCaseOutputs()` 已合并 plugin outputs
- 但 MCP 的 `toolGetCaseOutputs` 没有合并 plugin outputs
- Agent 的 DeployAgentSystemPrompt 没有提到插件系统

### 问题 3：get_ssh_info 只返回 1 个 IP

**影响链**：get_case_outputs 格式不可读 → Agent 转用 get_ssh_info → 只拿到每个 case 的第 1 台 IP → clash 配置 2 台而非 4 台。

## 修复方案

### Fix 1：正确解析 OutputMeta.Value

修改 `mod/mcp/mcp.go` 的 `toolGetCaseOutputs`：

```go
for k, meta := range outputs {
    var val interface{}
    if jsonErr := json.Unmarshal(meta.Value, &val); jsonErr != nil {
        result += fmt.Sprintf("| %s | %s |\n", k, string(meta.Value))
    } else {
        switch v := val.(type) {
        case string:
            // 处理相对路径转绝对路径
            result += fmt.Sprintf("| %s | %s |\n", k, v)
        case []interface{}:
            // 数组值（如 IP 列表）：逗号分隔显示
            items := make([]string, 0, len(v))
            for _, item := range v {
                items = append(items, fmt.Sprintf("%v", item))
            }
            result += fmt.Sprintf("| %s | %s |\n", k, strings.Join(items, ", "))
        default:
            result += fmt.Sprintf("| %s | %v |\n", k, v)
        }
    }
}
```

修复后输出：
```
| ecs_ip | 39.102.103.129, 39.102.84.182 |
| ssh_user | root |
```

### Fix 2：合并插件输出

在 `toolGetCaseOutputs` 末尾添加 plugin outputs 读取：

```go
if pluginOutputs := plugin.LoadPluginOutputs(c.Path); pluginOutputs != nil {
    result += "\n# Plugin Outputs\n\n"
    result += "| Key | Value |\n|-----|-------|\n"
    for k, v := range pluginOutputs {
        result += fmt.Sprintf("| %s | %s |\n", k, v)
    }
}
```

修复后 Agent 可看到：
```
# Plugin Outputs
| clash_config_file | /path/to/case/config.yaml |
| clash_node_count | 4 |
```

### Fix 3：Prompt 插件感知

在 `DeployAgentSystemPrompt` 新增：

1. **第 5 步更新**：说明 `get_case_outputs` 返回格式化的 IP 列表（逗号分隔），node>1 时必须用 get_case_outputs 而非 get_ssh_info
2. **新增第 5.5 步**：检查插件输出，proxy 模板自动生成 clash 配置，不要手动组装
3. **Compose 工作流**：步骤 6 新增调用 get_case_outputs 获取 IP 和插件输出

### Fix 4：工具描述更新

`get_case_outputs` 的 Description 更新为包含 plugin outputs 说明。

3.53 修复了 compose_up 异步导致的重复创建问题（量的问题），本次修复的是 output 不可读导致的信息丢失问题（质的问题）。两者共同确保 Agent 在多节点 proxy 场景下正确获取所有 IP 并使用已有的 clash 配置插件。
