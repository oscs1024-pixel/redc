# 3.48 AI Agent Instruction Following 增强

## 概述

本设计为 RedC AI Agent 增加 **Instruction Following** 核心能力，使 Agent 能够准确理解和拆解复杂多步指令，结构化地规划和追踪执行进度，并在失败时自动反思和纠错。

**涉及版本**：v3.1.9
**涉及文件**：
- `mod/ai/prompt.go` — 系统提示词
- `mod/mcp/mcp.go` — MCP 工具定义和执行
- `app_ai_chat.go` — Agent 循环和路由
- `mod/memory.go` — 记忆持久化
- `frontend/src/components/AI/AIChat.svelte` — 前端 UI

---

## 1. 架构设计

### 1.1 整体方案

**核心理念**：不改变 Agent Loop 架构，通过 **Prompt Engineering + 轻量工具 + 前端可视化** 增强 Instruction Following 能力。

```
用户指令
    │
    ▼
┌─────────────────────────────────┐
│  SmartAgentChatStream（入口）     │
│  ├─ classifyIntent() 意图分类    │
│  │   ├─ "deploy"  → DeployAgent │
│  │   ├─ "troubleshoot" → TroubleshootAgent │
│  │   └─ "ops"     → AgentOps   │
│  └─ runAgentLoop(prompt, tools) │
└─────────────────────────────────┘
    │
    ▼ (每轮循环)
┌─────────────────────────────────┐
│  TPAOR 结构化推理               │
│  1. Think  — 分析当前状态       │
│  2. Plan   — 调用 update_plan   │
│  3. Act    — 调用业务工具       │
│  4. Observe — 分析工具结果      │
│  5. Reflect — 决定下一步        │
│     ├─ 成功 → 更新 plan 进度    │
│     ├─ 失败 → 重试/换方案       │
│     └─ 新发现 → 调整 plan       │
└─────────────────────────────────┘
    │
    ▼ (完成时)
┌─────────────────────────────────┐
│  Memory 提取                    │
│  ├─ lesson: 通用经验            │
│  ├─ preference: 用户偏好        │
│  └─ failure: 结构化失败模式     │
└─────────────────────────────────┘
```

### 1.2 设计原则

1. **Prompt 驱动**：核心能力通过 System Prompt 引导 LLM 自主实现，不硬编码推理逻辑
2. **工具辅助**：`update_plan` 工具让 Agent 的计划对用户可见、可追踪
3. **最小改动**：复用现有 `runAgentLoop` 架构，只增加特殊工具处理和路由层
4. **渐进增强**：即使 LLM 不完美遵循 TPAOR，系统仍正常工作（优雅降级）

## 2. TPAOR 结构化推理框架

### 2.1 框架定义

TPAOR = Think → Plan → Act → Observe → Reflect

在 `AgentSystemPrompt` 中注入如下指令（适用于所有 Agent 模式）：

```
## 行动框架（每轮严格遵循 TPAOR）

1. **Think（思考）**：分析当前状态、用户目标、已知/缺失信息
2. **Plan（计划）**：
   - 首次：将任务拆解为编号步骤，调用 update_plan 展示
   - 后续：标注当前第几步，是否需要调整
3. **Act（行动）**：调用工具执行当前步骤
4. **Observe（观察）**：分析工具结果，是否符合预期
5. **Reflect（反思）**：
   - 成功 → 推进下一步，更新 plan
   - 失败 → 分析原因，重试/换方案/ask_user
   - 新发现 → 评估是否调整计划
```

### 2.2 复杂指令拆解

Prompt 中专门引导多目标指令的处理：

```
当用户指令涉及多个目标或步骤时：
1. 拆解为独立子任务
2. 分析子任务间的依赖关系
3. 调用 update_plan 展示完整计划
4. 逐个执行，每完成一个更新进度
```

### 2.3 错误自纠错

```
工具执行失败时：
1. 不立即放弃，先分析失败原因
2. 参数问题 → 修正参数重试（最多 2 次）
3. 环境问题 → 尝试替代方案
4. 权限/配额问题 → ask_user 求助
5. 更新 plan 中对应步骤为 failed
```

## 3. update_plan 工具

### 3.1 工具定义

```go
// mod/mcp/mcp.go
{
    Name: "update_plan",
    Description: "Update and display the current execution plan to the user...",
    InputSchema: {
        Properties: {
            "title":        string  — 计划标题
            "steps":        array   — [{name, status, detail}]
            "current_step": integer — 当前步骤索引 (0-based)
        },
        Required: ["steps"]
    }
}
```

**步骤状态值**：`pending` | `running` | `done` | `failed` | `skipped`

### 3.2 Agent Loop 中的特殊处理

`update_plan` 在 `app_ai_chat.go` 的 agent loop 中被特殊处理（类似 `ask_user`）：

```go
case "update_plan":
    // 1. Emit 前端事件
    a.emitEvent("ai-agent-plan", {
        conversationId, title, steps, currentStep
    })
    // 2. 持久化到 agent_tasks 表
    a.memoryStore.SaveTaskPlan(project, conversationId, title, planJSON)
    // 3. 返回确认
    resultContent = "Plan updated and displayed to user."
```

**关键**：`update_plan` 不会通过 MCP `executeTool` 走普通工具路径，而是在 agent loop 中拦截处理，因为它需要：
- emit 前端事件（MCP server 无法访问 Wails runtime）
- 持久化到 memory store（MCP server 无 memoryStore 引用）

### 3.3 前端渲染

**实时 Plan 卡片**（streaming 期间）：
```
┌─ 📋 部署 nginx 到阿里云 ──────────┐
│ ✅ 1. 查找现有场景                  │
│ ✅ 2. 选择模板                      │
│ ▶️ 3. 创建并启动场景                │
│ ⬚ 4. 等待实例就绪                   │
│ ⬚ 5. 安装 nginx                    │
│ ████░░░░░░ 2/5 (40%)               │
└─────────────────────────────────────┘
```

前端状态：
```js
let agentPlan = $state(null); // { title, steps, currentStep }
```

事件监听：
```js
EventsOn('ai-agent-plan', (data) => {
    agentPlan = { title, steps, currentStep };
});
```

Plan 在消息完成时被存入 `msg.plan`，历史消息中也能渲染。

## 4. 多 Agent 路由（意图识别）

### 4.1 SmartAgentChatStream

Agent 模式入口从 `AgentChatStream` 改为 `SmartAgentChatStream`，自动分类后路由：

```go
func (a *App) SmartAgentChatStream(convId string, msgs []AIChatMessage) error {
    intent := a.classifyIntent(msgs)
    switch intent {
    case "deploy":
        return a.DeployAgentChatStream(convId, msgs)
    case "troubleshoot":
        return a.TroubleshootAgentChatStream(convId, msgs)
    default:
        return a.AgentChatStream(convId, msgs)
    }
}
```

### 4.2 意图分类

使用轻量 LLM 调用（10s 超时），分析最后一条 user 消息：

```go
func (a *App) classifyIntent(msgs []AIChatMessage) string {
    // 取最后一条用户消息（截断 500 字符）
    // 发送 IntentClassificationPrompt
    // 解析结果：deploy / troubleshoot / ops
    // 失败默认返回 "ops"
}
```

**IntentClassificationPrompt**：
```
根据用户最后一条消息，判断意图类别。只输出一个单词。
- deploy: 部署/安装/搭建
- troubleshoot: 报错/失败/修复
- ops: 其他运维操作
```

### 4.3 Agent Prompts 对照

| Agent | Prompt | 默认轮次 | 超时 | 适用场景 |
|-------|--------|---------|------|---------|
| AgentOps | `AgentSystemPrompt` | 50 | 10min | 场景管理、模板操作、配置查看 |
| DeployAgent | `DeployAgentSystemPrompt` | 50 | 15min | 软件部署、模板生成、Compose 编排 |
| TroubleshootAgent | `TroubleshootAgentSystemPrompt` | 30 | 10min | 排错、日志分析、修复 |

### 4.4 TroubleshootAgent 设计

5 步固定 workflow（在 prompt 中定义）：

1. **收集信息** — list_cases + get_case_status + get_case_outputs
2. **复现问题** — exec_command 检查日志/服务状态
3. **定位根因** — 分析是 Terraform/API/网络/依赖/资源哪类问题
4. **修复尝试** — 提出方案、必要时 ask_user 确认、执行修复
5. **验证修复** — 重新检查服务状态

## 5. Memory 增强

### 5.1 任务记忆（agent_tasks 表）

```sql
CREATE TABLE IF NOT EXISTS agent_tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project TEXT NOT NULL,
    conversation_id TEXT NOT NULL,
    task_title TEXT NOT NULL,
    task_status TEXT DEFAULT 'in_progress',  -- in_progress / completed / failed
    plan_json TEXT,          -- 完整的 {title, steps, current_step} JSON
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**写入时机**：每次 Agent 调用 `update_plan` 时 upsert（按 conversation_id）
**完成标记**：Agent loop 正常结束（无 tool_calls）时调用 `CompleteTask(conversationId)`

### 5.2 失败模式记忆

Memory extraction prompt 新增 `failure` 类别：

```
failure|[tool:exec_command][error:apt lock][solution:等待 30-60 秒或 kill cloud-init]
failure|[tool:plan_case][error:quota exceeded][solution:切换区域或申请配额]
```

注入 prompt 时独立分区展示：

```
### 已知失败模式（遇到类似问题时直接套用解决方案）
- [tool:exec_command][error:apt lock][solution:等待 30-60 秒或 kill cloud-init]
```

### 5.3 Memory 数据流

```
Agent 对话结束
    │
    ▼
extractMemories()
    ├─ 构建对话摘要（user + assistant + 失败的 tool 消息）
    ├─ 发送 memoryExtractionPrompt 给 LLM
    └─ 解析结果，写入 agent_memory 表
        ├─ lesson|...
        ├─ preference|...
        └─ failure|[tool:...][error:...][solution:...]

下次对话开始
    │
    ▼
GetMemoryContext(project)
    ├─ 读取 agent_memory
    └─ 注入 system prompt
        ├─ ### 用户偏好
        ├─ ### 历史经验教训
        └─ ### 已知失败模式
```

## 6. 前端变更

### 6.1 新增状态

```js
let agentPlan = $state(null);  // { title, steps: [{name, status, detail}], currentStep }
```

### 6.2 新增事件

```js
EventsOn('ai-agent-plan', handler);   // 监听计划更新
EventsOff('ai-agent-plan');            // 清理
```

### 6.3 Plan 卡片渲染位置

- **实时**：在 streaming 区域，tool call 卡片上方
- **历史**：在历史消息的 assistant 气泡中，tool call 卡片上方
- 卡片包含：标题、步骤列表（状态图标+文字）、进度条

### 6.4 Agent 模式路由

```js
// 原来
await AgentChatStream(convId, chatMessages);
// 现在
await SmartAgentChatStream(convId, chatMessages);
```

### 6.5 重置逻辑

`agentPlan = null` 在以下时机重置：
- `ai-chat-complete` 事件
- 新建对话
- 切换对话
- 发送新消息

## 8. 维护注意事项

### 8.1 新增 Agent 模式

1. 在 `mod/ai/prompt.go` 添加新的 `XxxAgentSystemPrompt`
2. 在 `app_ai_chat.go` 添加 `XxxAgentChatStream` 方法
3. 在 `classifyIntent()` 的 `IntentClassificationPrompt` 和 switch 中添加新类别
4. 更新 Wails 绑定
5. 如需前端手动选择（而非自动路由），在 `modes` 数组中添加新模式

### 8.2 新增工具

1. 在 `mod/mcp/mcp.go` 的 `getTools()` 中添加工具定义
2. 在 `executeTool()` 的 switch 中添加执行逻辑
3. 如果工具需要特殊处理（如 ask_user、update_plan），在 `app_ai_chat.go` 的 agent loop 中添加 `else if` 分支
4. 在前端 `toolNameMap` 中添加中文显示名
5. 在相关 Agent 的 system prompt 中提及工具

### 8.3 调整 TPAOR 行为

- TPAOR 完全由 prompt 控制，修改 `AgentSystemPrompt` 中的"行动框架"部分即可
- 不同 Agent 可以有不同的推理框架（如 DeployAgent 的 6 步 workflow + TPAOR）
- 如果 LLM 不遵循 TPAOR，不会导致系统错误，只是缺少结构化推理输出

### 8.4 Memory 系统

- `agent_memory` 表存经验（lesson/preference/failure），上限 50 条/项目
- `agent_tasks` 表存任务计划，按 conversation_id 唯一
- Memory 提取是异步的（`go a.extractMemories(...)`），不阻塞主流程
- 新增 memory 类别需同时修改：提取 prompt → 解析代码 → GetMemoryContext 输出

### 8.5 意图分类注意

- `classifyIntent` 调用消耗一次 LLM API，增加 ~1-3 秒延迟
- 10 秒超时，失败默认走 ops Agent
- 只分析最后一条 user 消息（截断 500 字符），不分析对话上下文
- 如需更精确的分类，可增加关键词预判（避免 LLM 调用）
