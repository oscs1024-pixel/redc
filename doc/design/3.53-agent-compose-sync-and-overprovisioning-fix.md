# 3.53 Agent Compose 同步化 & 过量创建修复

## 问题背景

用户让 Agent "在 2 个云各开 2 台代理机器"，预期 4 台 VM，但实际创建了 12 台。

### 根因分析

**问题 1：compose_up 异步返回（主因）**

`app_compose.go` 的 `ComposeUp()` 使用 `go func()` 异步执行 `compose.RunComposeUp()`，MCP 工具 `toolComposeUp` 调用后立即返回 `"Compose deployment started"`。

时序问题：
1. Agent 调用 compose_up → 立即返回 "started"
2. Agent 调用 list_cases → 此时 compose 还在部署阿里云 case，AWS 的还没开始
3. Agent 看不到 AWS case → 误以为 AWS 没部署 → 手动 plan_case + start_case 创建 AWS case
4. compose 继续执行，也创建了 AWS case → **重复创建**

**问题 2：proxy 模板 node 参数语义不明**

`aliyun/proxy`、`aws/proxy` 模板有 `node` 变量（默认 10），一个 case 内创建 `node` 台 VM。
但 Agent 不了解此语义，将"2台机器"理解为"创建2个case"而非"1个case设node=2"。

结果：用户要 4 VM → Agent 认为 8 VM → 实际因 compose 重复变成 12 VM。

## 修复方案

### Fix 1：compose_up/compose_down MCP 同步化

**策略**：GUI 按钮保持异步（UI 不阻塞），MCP/Agent 调用走同步路径。

新增方法：
- `App.ComposeUpSync()` — 同步执行 `compose.RunComposeUpWithResult()`，返回部署结果
- `App.ComposeDownSync()` — 同步执行 `compose.RunComposeDown()`
- `AppBridge.MCPComposeUpSync()` / `MCPComposeDownSync()` — MCP 接口

代码变更：

```
mod/compose/compose.go:
  - RunComposeUp() → 保持原接口（调用 RunComposeUpWithResult 丢弃结果）
  + RunComposeUpWithResult() → 返回 *ComposeUpResult（含所有 service 的 case_id）
  + ComposeUpResult / ComposeUpService 类型

app_compose.go:
  - ComposeUp() → 仅 GUI 使用（异步 go func）
  + ComposeUpSync() → MCP 使用（同步阻塞，返回 case 列表）
  + ComposeDownSync() → MCP 使用（同步阻塞）

app_mcp_bridge.go:
  + MCPComposeUpSync() → 调用 ComposeUpSync
  + MCPComposeDownSync() → 调用 ComposeDownSync

mod/mcp/bridge.go:
  + MCPComposeUpSync() 接口
  + MCPComposeDownSync() 接口

mod/mcp/tools_compose.go:
  - toolComposeUp() → 改为调用 MCPComposeUpSync，返回 JSON 部署结果
  - toolComposeDown() → 改为调用 MCPComposeDownSync
```

### Fix 2：compose_up 返回 case ID

`RunComposeUpWithResult()` 返回：
```json
{
  "services": [
    {"name": "proxy_aws_1", "template": "aws/proxy", "case_id": "abc123...", "status": "deployed"},
    {"name": "proxy_aliyun_1", "template": "aliyun/proxy", "case_id": "def456...", "status": "deployed"}
  ]
}
```

tool 结果追加提示：
```
IMPORTANT: All cases listed above have been created by compose.
Do NOT manually create additional cases with plan_case/start_case for these services.
```

### Fix 3：proxy 模板 node 语义补充

1. **plan_case env 参数描述**（mcp.go）：
   > for proxy templates (aliyun/proxy, aws/proxy), the 'node' variable controls how many VMs to create in one case (default: 10). Set node=N for N machines rather than creating N separate cases.

2. **DeployAgentSystemPrompt 第4步**：
   > proxy 模板的 node 参数控制每个 case 创建的 VM 数量。用户说"N 台机器"时，正确做法是创建 1 个 case 并设 node=N，而不是创建 N 个 case。

3. **Compose 工作流提示词**：
   - compose_up 描述更新为"同步阻塞，返回 case ID"
   - 新增注意事项：compose_up 完成后不要再手动 plan_case/start_case

## 设计决策

1. **为什么不直接改 ComposeUp 为同步？**
   GUI 按钮调用 ComposeUp 后需要立即返回，否则 UI 会卡住。所以保留异步的 ComposeUp 给 GUI 用，新增同步的 ComposeUpSync 给 MCP/Agent 用。

2. **为什么在 tool result 里加防重复提示？**
   即使 prompt 里写了不要重复创建，有些模型在长对话中可能忘记指令。在 tool result 里再次提醒是最后一道防线。

3. **为什么同时修了 compose_down？**
   compose_down 有完全相同的异步问题。虽然用户报的 bug 是 compose_up，但 compose_down 不同步也会导致 Agent 误以为资源已销毁而继续操作。

## 潜在问题分析

分析了其他 MCP 工具，未发现类似异步问题：
- `toolStartCase` / `toolStopCase` — 直接调用 `c.TfApply()` / `c.Stop()`，同步
- `toolExecCommand` — 同步 SSH 执行
- `toolPlanCase` — 同步 `project.CaseCreate()`
- `toolComposePreview` — 同步

唯一发现的同类问题就是 compose_up 和 compose_down，已在本次修复中全部处理。
