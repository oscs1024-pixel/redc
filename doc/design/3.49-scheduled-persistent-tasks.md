# 3.49 AI Agent 定时与持续性任务增强

## 概述

增强 AI Agent 的时间感知和定时任务能力，使 Agent 能理解自然语言时间表达、创建周期性调度任务、并在长任务中断后恢复上下文。

## 问题背景

RedC 已有完整的 TaskScheduler（SQLite 持久化、10 秒轮询、周期重复、SSH 命令执行、通知），MCP 工具层也暴露了 `schedule_task`/`list_scheduled_tasks`/`cancel_scheduled_task`。但存在以下 5 个问题：

1. **Agent 无时间感知** — 不知当前时间，无法解析"1小时后"等相对时间
2. **Agent 不知如何使用调度工具** — prompt 中未引导 Agent 使用定时任务相关工具
3. **schedule_task 能力不足** — MCP 只暴露基础参数（case_id/action/scheduled_at），不支持周期/SSH命令/通知
4. **长任务无断点恢复** — Agent 超时后无法续接
5. **调度器不支持 kill 动作** — 无法定时强制销毁场景

## 技术方案

### 1. get_current_time 工具（时间感知）

**文件**: `mod/mcp/tools_scheduler.go`

新增 `get_current_time` 工具，返回：
```json
{
  "current_time": "2026-03-17T09:52:15+08:00",
  "local_time": "2026-03-17 09:52:15",
  "timezone": "CST",
  "utc_offset": "+8:00",
  "unix_timestamp": 1773988335
}
```

Agent 先调用此工具获取当前时间和时区，再根据用户的自然语言描述计算目标时间。

### 2. schedule_task 工具增强

**文件**: `mod/mcp/tools_scheduler.go`

原参数：`case_id`, `action(start/stop/kill)`, `scheduled_at`

新增参数：
| 参数 | 类型 | 说明 |
|------|------|------|
| `repeat_type` | string | 重复类型：once/daily/weekly/interval（默认 once） |
| `repeat_interval` | number | 间隔分钟数（仅 interval 类型） |
| `ssh_command` | string | SSH 命令（仅 action=ssh_command 时） |
| `notify` | boolean | 执行完成后发送系统通知 |

action 新增 `ssh_command`（在场景服务器上执行命令）和 `kill`（强制销毁）。

### 3. AppBridge 接口扩展

**文件**: `mod/mcp/bridge.go`, `app_mcp_bridge.go`

新增 `MCPScheduleTaskFull` 方法，MCP 工具层直接调用 `ScheduleTaskFull`，透传所有参数。

### 4. Agent Prompt 定时任务引导

**文件**: `mod/ai/prompt.go` — `AgentSystemPrompt`

在能力列表新增**定时任务**条目，新增完整的「定时与持续性任务」章节：
- 操作步骤：先获取时间 → 计算 RFC3339 → 调用 schedule_task
- 常见场景示例：定时关机、周期巡检、定时备份
- repeat_type 选择指南

### 5. 时间上下文注入

**文件**: `app_ai_chat.go` — `runAgentLoop`

系统提示末尾自动追加：
```
## 当前时间
当前时间: 2026-03-17 09:52:15 (时区: CST, UTC+8:00)
```

Agent 即使不调用 get_current_time 也能感知当前时间。get_current_time 在需要精确计算时间差时仍可用。

### 6. 未完成任务断点恢复

**文件**: `app_ai_chat.go` — `runAgentLoop`

从 `agent_tasks` 表查询最近 3 条状态为 `in_progress` 的任务，注入到系统提示：
```
## 未完成任务（可能需要续接）
以下任务之前未完成，如用户提到继续或恢复，可参考其上下文：
- [abc12345] 部署 Nginx 到 AWS (更新于 2026-03-17 08:30:00)
```

Agent 可据此理解用户的"继续上次的任务"类指令。

### 7. 调度器 kill 动作

**文件**: `mod/scheduler.go` — `AddTaskFull`

合法 action 列表从 `start/stop/ssh_command/auto_stop` 扩展为 `start/stop/kill/ssh_command/auto_stop`。`kill` 走 `default` 分支调用 `onExecute("kill")`。

## 数据流

```
用户: "1小时后关闭场景 ABC"
  ↓
Agent (prompt 中已有当前时间 09:52)
  ↓ 调用 get_current_time（可选，需精确时）
  ↓ 计算 10:52 → RFC3339 "2026-03-17T10:52:00+08:00"
  ↓ 调用 schedule_task(case_id=xxx, action="stop", scheduled_at="2026-03-17T10:52:00+08:00")
  ↓
MCP ExecuteTool → toolScheduleTask → MCPScheduleTaskFull → ScheduleTaskFull → AddTaskFull
  ↓
TaskScheduler 写入 SQLite，10秒轮询到时执行
```

## 与已有系统的关系

- **TaskScheduler** (`mod/scheduler.go`): 底层调度引擎不变，仅放宽 action 校验
- **MCP Server** (`mod/mcp/`): 工具层增强，保持向后兼容（新参数均可选）
- **Agent Memory** (`mod/memory.go`): 复用 agent_tasks 表的 in_progress 状态做断点恢复
- **Notification** (`notification.go`): 调度任务的 notify 走 TaskScheduler 已有通知回调

## 维护注意事项

1. `get_current_time` 无参数无副作用，是最安全的工具
2. `schedule_task` 的 `repeat_type` 默认 "once"，空值自动填充
3. 时间注入在 `runAgentLoop` 中，所有 Agent 模式（ops/deploy/troubleshoot）共享
4. 未完成任务查询限制 3 条，避免 prompt 膨胀
5. 断点恢复是"提示级"的（注入上下文），不是"状态级"的（不会自动重跑）
