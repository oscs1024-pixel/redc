# 3.27 资源拓扑预览（Plan Topology Visualization）

## 概述

资源拓扑预览功能允许用户在 **预定义场景** 和 **自定义部署** 页面中，对已创建但未启动的部署查看 Terraform Plan 的资源依赖关系拓扑图。通过解析 `terraform show` 获取资源变更列表，结合 `terraform graph` 获取 DOT 格式依赖关系，使用 ELK（Eclipse Layout Kernel）在前端自动计算分层布局，最终以 SVG 呈现交互式拓扑图。

功能亮点：
- **交互式缩放**：支持按钮和鼠标滚轮缩放（30%–300%）
- **资源详情**：拓扑节点显示关键配置（实例类型、CIDR、安全组端口规则等）
- **资源汇总表**：按资源类型统计数量并展示关键信息

## 关键技术细节

### 1. 后端数据聚合

`buildPlanPreview(workDir)` 是共享的核心函数，被 `GetCasePlanPreview()` 和 `GetDeploymentPlanPreview()` 调用：

```go
func (a *App) buildPlanPreview(workDir string) (*PlanPreview, error) {
    // 1. 解析 plan 文件获取资源变更
    resourceChanges, _ := te.GetPlanResourceChanges(ctx)
    
    // 2. 提取每个资源的关键配置详情
    for _, rc := range resourceChanges {
        detail := extractResourceDetail(rc)
        // detail 包含 instance_type, cidr_block, 安全组规则等
    }
    
    // 3. 按类型汇总资源数量
    typeSummary := buildTypeSummary(resources)
    
    // 4. 获取依赖图（DOT 格式）
    dot, _ := te.GetGraph(ctx)
    preview.Edges = parseDOTEdges(dot)
    
    return preview, nil
}
```

### 2. 资源详情提取

`extractResourceDetail()` 从 `terraform-json` 的 `Change.After`（`map[string]interface{}`）中提取关键配置值：

| 资源类型 | 提取字段 | 示例值 |
|---------|---------|--------|
| 计算实例（ECS/EC2/CVM） | `instance_type` | `ecs.t6-c1m1.large` |
| 计算实例 | `image_id` / `ami` | `ubuntu_22_04_x64_20G` |
| VPC / 子网 | `cidr_block` | `172.16.0.0/16` |
| 安全组规则（阿里云） | `port_range` + `ip_protocol` + `policy` + `cidr_ip` | `22/22 tcp allow 0.0.0.0/0` |
| 安全组（AWS） | `ingress` / `egress` 嵌套规则 | `ingress: tcp 22-22 [0.0.0.0/0]` |

**AWS 安全组特殊处理**：AWS 的 `aws_security_group` 资源将规则嵌套在 `ingress`/`egress` 数组中，`formatSGRules()` 负责解析并格式化：


### 3. 类型汇总

`PlanTypeSummary` 按资源类型汇总，保持首次出现顺序：

```go
type PlanTypeSummary struct {
    Type  string `json:"type"`   // 原始类型，如 alicloud_instance
    Label string `json:"label"`  // 可读标签，如 "实例"
    Count int    `json:"count"`
}
```

`humanizeResourceType()` 提供通用的可读化映射（覆盖阿里云、AWS、腾讯云等主流资源类型）。

### 4. DOT 节点名标准化

`terraform graph` 在不同模式下输出格式不同：

| 模式 | 节点名格式 | 示例 |
|------|-----------|------|
| `terraform graph`（无参） | 干净名称 | `"alicloud_instance.instance"` |
| `terraform graph -plan=case.tfplan` | `[root]` 前缀 + `(expand)` 后缀 | `"[root] alicloud_instance.instance (expand)"` |
| plan 模式还包含 | provider 节点 | `"[root] provider[\"registry...\"]"` |
| plan 模式还包含 | 变量/输出节点 | `"[root] var.instance_name"`, `"[root] output.ecs_ip (expand)"` |

`normalizeDOTNode()` 统一剥离 `[root] ` 前缀和 ` (expand)` / ` (close)` 后缀：

```go
func normalizeDOTNode(name string) string {
    name = strings.TrimPrefix(name, "[root] ")
    name = strings.TrimSuffix(name, " (expand)")
    name = strings.TrimSuffix(name, " (close)")
    return name
}
```

前端通过 `addrSet`（plan 中的资源地址集合）过滤边，自动排除 provider、variable、output、local 等非资源节点的边。

### 5. ELK 布局配置

```javascript
import ELK from 'elkjs/lib/elk.bundled.js';  // 静态导入，浏览器安全

const graph = {
  id: 'root',
  layoutOptions: {
    'elk.algorithm': 'layered',              // 分层算法（适合 DAG）
    'elk.direction': 'DOWN',                 // 从上到下
    'elk.spacing.nodeNode': nodeSpacing,      // 动态：>8 节点用 25，否则 35
    'elk.layered.spacing.nodeNodeBetweenLayers': layerSpacing,  // 层间距
    'elk.padding': '[top=15,left=15,bottom=15,right=15]',
    'elk.layered.nodePlacement.strategy': 'BRANDES_KOEPF',  // 紧凑节点放置
  },
  children: [...],  // 节点：{ id: address, width: 180, height: 48 }
  edges: [...],     // 边：{ sources: [from], targets: [to] }
};

const layout = await elk.layout(graph);
// 返回 { children: [{x, y, width, height}], edges: [{sections}] }
```

**动态间距策略**：当节点数超过 8 个时自动缩小节点间距和层间距，保持拓扑图紧凑。

### 6. SVG 渲染

- **节点**：圆角矩形，默认 180×48，有详情时扩展为 180×58
  - 第一行：资源类型标签（左）+ 操作标记 `+`/`~`/`-`（右）
  - 第二行：资源名称（灰色）
  - 第三行（可选）：详情文本（斜体灰色，如 `ecs.t6-c1m1.large`）
  - 左侧圆点指示操作类型颜色
- **连线**：基于 ELK 返回的 `sections[0].startPoint` → `bendPoints` → `endPoint` 构建 SVG `<path>`，带箭头 marker
- **着色规则**：
  - 创建（create）= 绿色 `#10b981` 边框
  - 更新（update）= 蓝色 `#3b82f6` 边框
  - 删除（delete）= 红色 `#ef4444` 边框
  - 替换（delete+create）= 橙色 `#f59e0b` 边框
- **viewBox 自适应**：根据实际节点坐标计算，加 40px padding
- **缩放控制**：
  - `−` / `+` 按钮：±10% 步进
  - 鼠标滚轮：±8% 步进
  - 重置按钮：回到 100%
  - 范围：30%–300%
  - 实现：CSS `transform: scale()` + `transform-origin: center top`，外层 `overflow: auto`

### 7. 资源汇总表格

模态框中拓扑图上方展示资源类型汇总表：

| 列 | 内容 | 说明 |
|----|------|------|
| 资源类型 | 可读标签 + 原始类型名 | 如 "实例 `alicloud_instance`" |
| 数量 | 蓝色圆点（>1 时高亮） | 便于识别批量资源 |
| 关键信息 | 首个同类资源的详情值 | 如 `ecs.t6-c1m1.large · ubuntu_22_04` |

### 8. 降级策略

当 ELK 布局失败（`elkNodes` 为空数组）时，自动降级为简单资源列表：

```svelte

```

## 交互流程

### 预定义场景（Cases.svelte）

1. 用户在预定义场景页面，对 `created` 或 `stopped` 状态的场景点击 **🔍 预览** 按钮
2. 前端调用 `GetCasePlanPreview(caseId)`
3. 后端读取 `case.tfplan` + `terraform graph`，返回 `PlanPreview` JSON
4. 前端 `layoutTopology()` 使用 ELK 计算布局坐标
5. 弹出模态框（最大宽度 1100px，最大高度 90vh）展示：
   - 顶部统计栏：`+ 创建 5` `~ 更新 0` `- 删除 0` + 资源/依赖数量
   - 资源汇总表格：按类型统计数量和关键信息
   - 缩放控制栏：`−` / 百分比 / `+` / 重置
   - 中部 SVG 拓扑图（可滚动，最大高度 65vh，支持鼠标滚轮缩放）
   - 底部按钮：关闭 / 启动场景

### 自定义部署（CustomDeploymentList.svelte）

1. 用户在自定义部署页面，对 `pending` 或 `stopped` 状态的部署点击 **🔍 预览** 按钮
2. 前端调用 `GetDeploymentPlanPreview(deploymentId)`
3. 后端通过 `projectPath + deploymentId` 定位工作目录，执行相同的 `buildPlanPreview()` 流程
4. 模态框展示内容与预定义场景完全一致

## 注意事项

### elkjs 导入方式

必须使用 `elkjs/lib/elk.bundled.js`（浏览器安全版本），**不能** 使用默认入口 `elkjs`（即 `lib/main.js`），后者会 `require('web-worker')` 导致 Wails WebView 构建失败。

使用静态 `import` 而非动态 `import()`，确保 Vite/Rollup 在构建时正确处理 CJS → ESM 转换。

### terraform graph 兼容性

- 优先使用 `Graph(ctx, GraphPlan("case.tfplan"))`，失败时回退到 `Graph(ctx)`（无 plan 参数）
- Plan 模式输出包含 `[root]` 前缀，需要 `normalizeDOTNode()` 标准化
- `parseDOTEdges()` 使用 `seen` map 去重，避免重复边

### Wails 绑定

新增的两个后端函数需要在 **4 个绑定文件** 中同步注册（项目存在双层嵌套路径）：
- `frontend/wailsjs/go/main/App.js` + `App.d.ts`
- `frontend/wailsjs/wailsjs/go/main/App.js` + `App.d.ts`

### Svelte 5 响应式

`elkNodes`、`elkEdges`、`svgViewBox` 均为 `$state()` 变量。在 `layoutTopology()` 中先用局部变量计算完所有数据，最后一次性赋值触发 UI 更新。
