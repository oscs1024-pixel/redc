# 3.50 Agent 部署失败修复 — 架构/模板/输出溢出问题

## 背景

通过分析真实 Agent 部署执行记录（nuclei 安装失败案例），发现 Agent 在软件部署流程中存在 5 个系统性问题，导致简单工具安装任务失败。

## 问题分析

### 案例：在 AWS t4g（ARM64）上安装 nuclei

Agent 执行了 6 轮工具调用，最终未能成功安装 nuclei。

| 轮次 | 行为 | 问题 |
|------|------|------|
| 1 | wget 下载 nuclei linux-amd64 | 未检查目标机器架构，t4g 是 ARM64 却下载了 amd64 |
| 2 | 解压后执行 nuclei -version | "cannot execute binary file: Exec format error" |
| 3 | 尝试 apt install nuclei | 仓库中没有 nuclei |
| 4 | 尝试 go install | 系统无 Go 环境 |
| 5 | wget 下载 Go 安装包（带进度条） | 32KB 进度条输出占满 8000 字符缓冲区，有效输出被截断 |
| 6 | 继续尝试 go install nuclei | Go PATH 未配置，安装命令失败 |

### 5 个根因

1. **架构不敏感**：Agent 未在安装前执行 `uname -m` 确认 CPU 架构
2. **f8x 模板未利用**：f8x-bash 模板可安装 nuclei，但 Agent 不知道 f8x 包含哪些工具
3. **userdata 模板未优先**：DeployAgentSystemPrompt 虽然提到"优先使用 userdata 模板"，但是建议而非硬规则
4. **wget 进度条溢出**：`wget` 不加 `-q` 导致进度条输出占满结果缓冲区（maxToolResultLen=8000）
5. **dpkg lock 处理不当**：新实例 cloud-init 占锁，固定 `sleep 30` 可能不够

## 修复方案

### Fix 1：DeployAgentSystemPrompt 增强（mod/ai/prompt.go）

重构第 5 步"部署软件"：

```
第 5.0 步（必须）：确认目标机器 CPU 架构
- 通过 exec_command 执行 uname -m 获取架构
- 后续下载必须选择匹配架构的版本

第 5.1 步（必须）：优先检查 userdata 模板
- 渗透测试工具 → 优先 f8x-bash
  f8x 包含 nuclei/nmap/masscan/subfinder/httpx/ffuf/amass 等 100+ 工具
  用法：exec_userdata(template_name='f8x-bash') → ./f8x -nuclei
- 其他工具 → 检查 list_userdata_templates
- 只有无模板时才手动安装

手动安装时的额外规则：
- 下载文件必须用 wget -q 或 curl -sL
- Go 工具可用官方一键安装脚本（如 curl -sL nuclei.projectdiscovery.io/install | bash）
```

### Fix 2：exec_command 工具描述增强（mod/mcp/mcp.go）

在工具描述中追加：

> "IMPORTANT: When downloading files, always use 'wget -q' or 'curl -sL' to suppress progress output which can overflow the result buffer"

### Fix 3：list_userdata_templates 工具描述增强（mod/mcp/mcp.go）

明确提及 f8x-bash：

> "IMPORTANT: f8x-bash template can install 100+ penetration testing tools (nuclei, nmap, masscan, subfinder, httpx, etc.). Always check this list BEFORE writing manual install commands."

### Fix 4：f8x-bash case.json 描述增强

从：
```json
"description": "f8x 渗透测试工具集，一键部署多种渗透测试工具"
```

改为：
```json
"description": "f8x 渗透测试工具集，一键部署 nuclei/nmap/masscan/subfinder/httpx/ffuf/amass/dirsearch/sqlmap/xray 等 100+ 安全工具"
```

### Fix 5：dpkg lock 等待策略优化

从固定 `sleep 30` 改为循环等待：
```bash
while fuser /var/lib/dpkg/lock-frontend >/dev/null 2>&1; do sleep 5; done
```

### Fix 6：AgentSystemPrompt 工作原则同步

非部署 Agent 也增加两条工作原则：
- 优先使用 userdata 模板（特别是 f8x-bash）
- 安装二进制文件前先确认 CPU 架构

## 修改文件

| 文件 | 修改内容 |
|------|----------|
| `mod/ai/prompt.go` | DeployAgentSystemPrompt 第 5 步重构 + dpkg lock 策略 + AgentSystemPrompt 工作原则 |
| `mod/mcp/mcp.go` | exec_command + list_userdata_templates 工具描述增强 |
| `templates/userdata-templates/f8x-bash/case.json` | description 增加具体工具列表 |

## 预期效果

修复后，Agent 遇到 "安装 nuclei" 请求时的理想行为：

1. `uname -m` → aarch64
2. `list_userdata_templates` → 发现 f8x-bash（描述中明确含 nuclei）
3. `exec_userdata(template_name='f8x-bash')` → 安装 f8x
4. `exec_command("./f8x -nuclei")` → 安装 nuclei（f8x 自动处理架构）
5. `exec_command("nuclei -version")` → 验证成功

从 6 轮失败 → 5 轮成功，核心改善是 Agent 知道了"有什么工具可用"以及"目标机器是什么架构"。

## 设计原则

- **知识注入而非行为约束**：告诉 Agent"f8x 包含 nuclei"比"必须用 f8x"更灵活
- **前置检查而非后置修复**：架构检查放在安装前，而非等到 exec format error 再纠正
- **抑制噪音而非扩大缓冲**：用 wget -q 从源头减少输出，比增大 maxToolResultLen 更高效
- **渐进式引导**：工具描述 → prompt 指令 → 模板元数据，三层叠加确保 Agent 不遗漏
